# ADR-001: ドキュメント運用方針

## Status
Accepted

## Date
2026-04-13

## Context

グリッド取引ボットの開発において、README やその他のドキュメントの品質が不安定だった。具体的には以下の問題が発生した：

- README に不自然な日本語表現が含まれていた
- 絵文字の過剰使用が Windows コマンドプロンプトでエンコーディングエラーを引き起こした
- 中国語（「歡迎です」）が混入していた
- 設定値と実際のコードの値が不一致だった

## Decision

ドキュメントの作成・更新には `documentation-and-adrs` スキルを使用する。以下のルールに従う：

1. **README は簡潔に**：プロジェクトの概要、セットアップ手順、コマンド一覧を記載
2. **日本語は自然な表現で**：助詞の不足、不自然な言い回しを避ける
3. **絵文字は使わない**：ターミナル環境でのエンコーディング問題を防ぐ
4. **設定値は実際のコードと一致させる**
5. **ADRs で設計判断の理由を記録する**

## Alternatives Considered

### ドキュメントなし
-  Pros: 作業コストゼロ
-  Cons: 新規参画者・将来の自分が混乱する、同じミスを繰り返す
-  Rejected: ドキュメントは長期視点で必ず必要になる

### 英語のみ
-  Pros: 国際標準、エンコーディング問題が少ない
-  Cons: ユーザー（日本人）にとって読みづらい
-  Rejected: 日本語メインで使用するため

## Consequences

- ドキュメント作成に追加コストがかかる
- 品質が安定し、メンテナンス性が向上する
- 将来の開発者・エージェントがプロジェクトを正しく理解できる
